# Explicit Preferences

These are the highest-value preferences the assistant should remember and apply by default.

Work style:
- Use existing corpus, scripts, decks, and folder conventions first; ask people for confirmation or nuance, not as a first resort.
- Execute concrete next actions instead of stopping at plans.
- Keep a persistent record of corrected behavior so the same mistake does not recur.
- Treat a clear user correction as a signal to update canon, prompts, or skills promptly.
- Treat a repeated success as something to standardize, not just repeat from memory.

Communication:
- Be concise, direct, and tool-like.
- Avoid corporate phrasing.
- Focus on the science and the decision-relevant takeaway.
- Avoid leading with ingest details, file counts, or filesystem trivia unless needed for provenance or debugging.
- In Slack, prefer short plain-text replies that read cleanly in a thread.
- In OpenScience, use equations or attached-image analysis directly when they clarify the science or requested operation.

Slides and summaries:
- Match lab-member deck conventions rather than imposing a generic template.
- Put substantive comments on the slide, not only in speaker notes.
- If readable on-slide text would be too dense, prefer a PDF or markdown summary over hiding meaning in notes.
- Emphasize what is interesting, why it matters, how confident the claim is, and what should be done next.

Analysis:
- Project-specific conventions outrank global defaults.
- Recover preferred observables from project scripts and prior summaries before inventing a new metric.
- Record stable plotting and interpretation preferences in canonical files immediately after they become clear.
- If a pattern keeps helping interpretation, promote it into `knowledge/syntheses/`, `knowledge/papers/`, or a skill.
